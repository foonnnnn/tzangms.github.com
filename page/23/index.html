我站在河岸留言舞台上

Jan 10th, 2009

這張照片對我來說還滿有意義的, 感謝叮咚幫我拍這張! 真的! 超感謝!

以前在學生時代玩團時, 就很想到台北表演, 想要到 the Wall、地下社會、海邊的卡夫卡、女巫店跟河岸留言, 這些聽獨立音樂的人都一定知道的地方表演, 終於我也站上河岸留言的舞台, 唱起自己的歌了。

雖然這對大部分在台北的樂團、音樂人來說似乎不是大不了的事。

只是台北跟台北以外的音樂資訊落差真的頗大, 當台北的音樂人都在唱著自己創作而不唱 copy 時, 在台北以外的地方, 卻呈現相反的狀況, 我在二技時唱著自己的創作, 每次得到的回應都是: “怎麼都唱一些沒聽過的歌” 不然就是 “我想聽周杰倫的雙節棍” …. 但是我還是這樣唱了好久, 唱到大家都會跟著我一起唱, 一起唱我的歌。

當時覺得在台北唱歌仍然是遙不可及的, 一直覺得程度差太多, 怎麼可能有辦法到這些地方唱。 不過當時就打定主意, 決定將來一定要到台北找工作, 在台北玩團, 這樣我一定也可以到這些地方唱我自己的歌。 只是等到鼓手去當兵退伍後, 貝斯手也準備要去當兵了, 我的樂團似乎怎麼也搭不起來。

最後誰知道我現在把時間都花在寫程式了。

幹, 下一個人生的目標就是 … 出第二張唱片, 然後把這些地方全都唱過一遍!

Twitthat 新增字數檢查

Jan 7th, 2009

[caption id=”attachment_2306” align=”alignleft” width=”300” caption=”twitthat word limit count function.”][/caption] 今天幫 twitthat 加上了字數檢查的功能, 這也是一個擱了很久的功能阿~ (茶)

因為常看到有些 twitthat 送出到 twitter 的訊息, 由於字數太多, 超過 140 個字, 而放在訊息最後面的網址都被截斷了, 所以加上這個功能, 用來提示還剩下幾個字, 避免訊息在 timeline 頁面被截斷。 話說 xdite 的 twitio.us 的做法就不賴, 把 user 另外想加上的文字放在最後面, 其實滿清楚的。 只是我在想 UI 要怎麼做會比較簡單 ~”~

為了維持簡單的界面, 基本上字數沒有超過 130 個字, 是不會看見這個提示的。 一直到訊息超過 130 個字, 提示才會出現, 提醒還有幾個字可以打, 避免網址又被截斷。

Twitthat UI Improvement

Jan 6th, 2009

[caption id=”attachment_2303” align=”alignleft” width=”300” caption=”twitthat UI improvement screenshot”][/caption] 其實覺得 twitthat 的 UI 頗刺眼, 所以今天就改了 twitthat 的 UI, 不過還真的是 UI 苦手 Orz

然後開始用一堆 css 圓角來取代先前的圖片圓角, 這樣應該會比較快, 不過… 這個 css 圓角完全不支援 ie (笑) 請換掉 ie, 好嗎?

ok 開始離題, 其實本來是想把 twitthat 整個搬到 Google App Engine 上去, 也寫一半了, 因為 public timeline 目前已經快要 20 萬筆, 所以整個變很慢, 然後我 SQL 又超弱, 所以想說搬過去, 用 datastore 應該就不會有這個問題。 不過後來想一下, 這剛好就是學習的好機會阿! 所以昨天就花時間開著 MySQL 來打 Query, 看看怎麼樣會比較快。

hmm, 使用者的 twits 是存放在 “bookmarks” 這個 table, 而先前取得 public timeline 的方式, 是用 “bookmarks” 來 join “users” 跟 “links” 這 2 個 table, 很笨對吧 XD 因為要取得網址跟使用者資料, 所以 join 了這兩個 table… anyway, 以前資料量小還沒感覺, 現在資料多, 感覺就很明顯了。 越後面的頁數, SQL 查詢的時間就越久, 要看 public timeline 的最後一頁時, 居然要花上 15 ~ 17 秒。

結果昨天就改用 sub query 的方式而不用 join, 原本要花上十幾秒的最後一頁, 變成只花了 0.3 秒 (咳) 真的是差太多了! 真的該好好學一下 SQL 的 Orz

Linode 更換 Data Center

Jan 5th, 2009

有買 Linode 的人就知道, 他們目前在美國有 4 個機房, 可以讓你選擇 VPS 要放的機房, 所以好像通常都會選加州的機房, 因為會比較快(?), 其實之前我就買過一次 Linode, 不過因為當時加州已經沒得放, 所以就選了德州 Dallas 的機房, 而這次買就是選加州的機房, ping time 還滿不賴的, 比起我手上有的主機都還要好。

不過今天傑洛大大問到我 Linode 的使用心得, 我回答說:「ping time 很不錯阿!」, 傑洛大就問說:「那實際速度呢?」後來才用 ftp 實際測一下上下傳速度, 沒想到… 我這台位於加州的 Linode, 上傳有 170k/s 但是下載居然只有 20k/s Orz 感謝傑洛大, 幫我上了一課。 就算 ping time 快, 但是檔案下載很慢也是於事無補阿 XD ( 應該是這樣吧? 有人可以來開導我一下嗎? )

後來我找到了這篇文章, 實際測試了一下下載的速度, 結果 dallas 跟 atlanta 機房的下載速度都有 200k/s 阿! 而我那個機房的下載只有 20k/s 真是糟糕 Orz
